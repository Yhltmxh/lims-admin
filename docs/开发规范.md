# **技术开发Claude规范**

本文分两部分：**第一部分**是本项目的强制规范（来自团队 CLAUDE.md，必须遵守）；**第二部分**是行业通用后端最佳实践（补充建议，结合本项目 Java / Spring / MyBatis-Plus / Feign / MQ 技术栈）。

 

# **第一部分 本项目强制规范**

## **一、开发流程（强制，每步必须执行对应 skill）**

### **新功能开发 9 步**

​            \1.     superpowers:brainstorming —— 讨论需求

​            \2.     superpowers:writing-plans —— 生成实现计划（含验收用例）

​            \3.     superpowers:test-driven-development —— 写代码前先写测试

​            \4.     superpowers:executing-plans + verification-before-completion —— 按 Phase 执行，每个 Phase verification 通过后立即 commit

​            \5.     superpowers:verification-before-completion —— 整体完成前验证

​            \6.     superpowers:requesting-code-review —— 代码审查（PR 必须带用例）

​            \7.     输出技术细节总结（DDL / 新增接口 / 新增 Job / 新增配置）→ docs/technical_decisions

​            \8.     输出学习总结 → Memory

​            \9.     /git-commit-push —— 提交

**军规一**：无用例不合并，用例先于代码存在。禁止口头说"按流程"而跳过 skill。

### **Bug 修复 7 步**

​            \1.     superpowers:systematic-debugging —— 分析根因，不直接改代码

​            \2.     superpowers:test-driven-development —— 写复现测试（FAIL）

​            \3.     修复代码 —— 让复现测试 PASS

​            \4.     回归验证 —— 编译 + 相关模块测试全部通过

​            \5.     superpowers:verification-before-completion

​            \6.     检查影响范围 —— grep 所有调用点

​            \7.     commit

**禁止**：出错后直接重试、不看日志就改代码、同时改多处。

## **二、Java 编码规范（项目统一）**

​            \1.     **Service 接口 + 实现类分离**：实现类放 impl/ 子包。

​            \2.     **MyBatis-Plus 必须用 LambdaQueryWrapper + 方法引用，禁止字符串字段名**：✅ .eq(MerchantPeriodRecord::getSettleAccountId, id)

​            a.     ❌ .eq("settle_account_id", id)

​            \3.     **import 顶部声明**，代码行不出现全限定类名。

​            \4.     **时间类型统一 Date**，禁用 LocalDate / YearMonth；日期操作用 cn.hutool.core.date.DateUtil。

​            \5.     **JSON 解析用 fastjson** com.alibaba.fastjson.JSON.parseObject()，不用 Jackson。

​            \6.     **ResponseBaseVo 用 ok()**，没有 success() 方法。

​            \7.     **RPC 接口返回 VO**，不暴露内部实体类（建专门的 ResVO）。

​            \8.     **Fallback Factory 用静态 RES 常量**：

 



```
private static final ResponseBaseVo RES = ResponseBaseVo.fail("501000000", "xxx-service exception! ");
```





​            \9.     **枚举判断禁用魔法数字**，用 AccountTypeEnum.ACCOUNT_TYPE_PERIOD.getTypeValue()。

​            \10.     **方法入参精简**，能查询得到的不让调用方传。

## **三、架构设计规范**

​            \1.     **新老模型路由放业务入口**（OrderServiceImpl 等），不放底层工具方法。老代码完全不动，新模型在业务入口 if-else 分流。

​            \2.     **下单冻结用 RPC 同步，不用 MQ**（强一致；失败时发 MQ cancel + user-service 幂等判断）。

​            \3.     **isNewModel 入参只传结算客户 ID**，不传 MerchantLimit / Order + OrderExtendInfo：下单场景：accountResVO.getSettleAccountId()

​            a.     退款场景：orderExtendInfoDao.getOrderExtendInfo(orderId).getSettlementId()

​            \4.     **查门店账期记录只用 merchantId**，不联动 settleAccountId（切换结算客户会导致绑错）。

​            \5.     **Feign Client 接口写在 client 模块**，Controller 和 Client 两处定义。

​            \6.     **跨服务依赖的 VO 放 entity 模块**。

## **四、灰度判断（新老模型）必须统一模式**

所有服务、所有场景判断新老模型必须走**同一链路**：

 



```
OpenAccountResVO accountResVO = userService.queryMerchantAndUserInfo(merchantId);
Integer modelType = accountResVO.getModelType();
```





禁止直接查 SettlementAccount 表判断。任何场景（下单 / 退款 / 查余额 / 汇总）都统一走 userService.queryMerchantAndUserInfo。

 

# **第二部分 通用后端开发规范（行业最佳实践补充）**

以下为业界通用规范，作为上面项目硬规则的补充，落地时以「第一部分」为准。

## **五、API 接口设计**

​                ● **统一响应结构**：复用项目的 ResponseBaseVo（code / message / data），不要每个接口自定义结构。

​                ● **入参校验前置**：用 @Validated + JSR-303 注解（@NotNull / @Size 等）在 Controller 层拦截非法参数，不把校验混进业务逻辑。

​                ● **接口幂等**：所有写接口（下单、支付、退款）必须可重复调用而结果一致，用唯一业务键 + 幂等表 / 去重判断。

​                ● **分页有上限**：pageSize 设硬上限（如 ≤ 200），禁止 select * 全表返回。

​                ● **接口版本与兼容**：对外接口字段只增不删；废弃字段先标注 @Deprecated 再灰度下线。

​                ● **避免 N+1 查询**：批量场景用 in / 批量接口，禁止循环里逐条 RPC / SQL。

## **六、异常处理与错误码**

​                ● **全局异常处理**：用 @RestControllerAdvice 统一兜底，避免堆栈直接抛给前端。

​                ● **错误码规范**：沿用项目错误码段（如 501xxxxxx），分「系统错误 / 业务错误」两类，业务错误给出可读 message。

​                ● **不吞异常**：catch 后必须记录日志或重新抛出，禁止空 catch。

​                ● **异常即回滚**：事务方法内抛出异常要确保触发回滚（注意 Spring 默认只对 RuntimeException 回滚）。

​                ● **降级返回兜底**：Feign / RPC 失败走 Fallback，返回项目约定的静态 RES 常量。

## **七、日志规范**

​                ● **关键链路打日志**：入口参数、外部调用出入参、异常分支都要有日志，便于排查。

​                ● **带上下文**：日志里带 traceId / 订单号 / 商户号等业务标识，能串起一次完整调用。

​                ● **级别正确**：ERROR 只记真正异常，INFO 记关键节点，调试信息用 DEBUG，禁止生产刷 INFO 大日志。

​                ● **禁打敏感信息**：密码、密钥、完整身份证 / 银行卡号脱敏后再打。

​                ● **禁止字符串拼接**：用占位符 log.info("order {} paid", orderId)，不要 + 拼接。

## **八、数据库与事务**

​                ● **事务范围最小化**：事务方法里不要放 RPC / HTTP / 发 MQ 等耗时操作，先落库再异步。

​                ● **大事务拆分**：批量写拆批提交，避免长事务锁表。

​                ● **索引意识**：高频查询字段建索引；LambdaQueryWrapper 条件要能命中索引，避免函数包裹字段导致索引失效。

​                ● **乐观锁防并发覆盖**：更新关键余额 / 状态用 version 字段或 update ... where status = 旧值。

​                ● **避免隐式类型转换**：字段类型与查询参数一致（如本项目时间统一 Date）。

## **九、并发与幂等**

​                ● **幂等三件套**：唯一键约束 + 去重表 / 状态机 + 重试可重入。

​                ● **状态机流转**：订单 / 支付状态用明确状态机，只允许合法跃迁，更新带前置状态条件。

​                ● **分布式锁**：跨实例的临界区用 Redis / 数据库锁，锁要有超时，避免死锁。

​                ● **MQ 消费幂等**：消费端按业务键去重（呼应项目"失败时发 MQ cancel + user-service 幂等判断"）。

## **十、缓存使用**

​                ● **缓存一致性**：更新数据时先更库再删缓存（Cache-Aside），避免脏读。

​                ● **防穿透 / 击穿 / 雪崩**：空值缓存、热点 key 加锁重建、过期时间加随机抖动。

​                ● **缓存只做加速**：缓存不可用时要能降级回源，不能因缓存挂了整个服务不可用。

## **十一、安全**

​                ● **越权校验**：每个接口校验当前用户是否有权操作目标资源（商户 / 订单归属）。

​                ● **SQL 注入防护**：用参数化查询 / MyBatis 占位符，禁止拼接 SQL。

​                ● **外部输入不可信**：金额、ID 等关键参数后端二次校验，不信任前端传值。

​                ● **敏感配置不入库不进仓库**：密钥走配置中心 / 环境变量。

## **十二、测试**

​                ● **TDD 先行**（呼应项目流程）：用例先于代码存在，无用例不合并。

​                ● **覆盖关键路径**：核心业务（下单 / 退款 / 结算）覆盖正常 + 异常 + 边界。

​                ● **复现测试**：Bug 修复必须先写一个能复现的失败测试，修完转绿。

​                ● **回归验证**：改动后编译 + 相关模块测试全部通过再提交。

## **十三、代码审查与提交**

​                ● **PR 必须带用例**（项目军规）。

​                ● **小步提交**：每个 Phase verification 通过后立即 commit，避免大杂烩 PR。

​                ● **审查关注点**：根因是否正确、影响范围是否 grep 全、是否符合项目编码 / 架构规范。

​                ● **技术决策留档**：DDL / 新增接口 / Job / 配置写入 docs/technical_decisions。

## **十四、性能优化**

注：团队原文档此处还有「性能优化模式」等章节未完整提供，以下为通用要点，待补充原文后并入。

​                ● **先量化再优化**：用监控 / 压测定位瓶颈，不靠猜。

​                ● **批量优于循环**：批量查询 / 批量写，消除 N+1。

​                ● **异步解耦**：非强一致的耗时操作（通知、统计）走 MQ / 异步线程池。

​                ● **连接池 / 线程池合理配置**：DB 连接、HTTP 客户端、线程池大小按压测结果设定。

​                ● **慢 SQL 治理**：开启慢查询日志，定期清理与加索引。
